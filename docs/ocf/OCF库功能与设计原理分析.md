# OCF库功能与设计原理分析

## 概述

**OCF (Open CAS Framework)** 是Intel开发的高性能块存储缓存元库，用C语言编写。它是一个完全平台和系统独立的框架，通过用户提供的环境包装层访问系统API。OCF与软件栈的其他部分紧密集成，提供高性能、低延迟的缓存功能。

### 核心特点

- **平台独立**: 通过环境抽象层（Environment Abstraction Layer）实现跨平台
- **高性能**: 优化的缓存算法和数据结构
- **低延迟**: 异步I/O处理和快速路径优化
- **灵活配置**: 支持多种缓存模式、清理策略、淘汰策略
- **可扩展**: 模块化设计，易于集成到现有系统

## 2. 缓存模式 (Cache Modes)

OCF支持6种缓存模式，每种模式有不同的写入策略：

### 2.1 Write-Through (WT) - 写透模式

**工作原理**:
- **读操作**: 
  - Cache Hit: 从缓存读取
  - Cache Miss: 从核心读取，并写入缓存
- **写操作**: 
  - 同时写入缓存和核心
  - 确保数据一致性

**特点**:
- ✅ 数据一致性最好
- ✅ 缓存故障不影响数据安全
- ❌ 写入延迟较高（需要等待核心写入完成）

**适用场景**:
- 对数据一致性要求极高的场景
- 写入性能要求不高的场景

**代码路径**: `engine_wt.c`

### 2.2 Write-Back (WB) - 写回模式

**工作原理**:
- **读操作**: 
  - Cache Hit: 从缓存读取
  - Cache Miss: 从核心读取，并写入缓存
- **写操作**: 
  - 只写入缓存，标记为dirty
  - 延迟写入核心（通过清理策略）

**特点**:
- ✅ 写入性能最好
- ✅ 写入延迟最低
- ❌ 需要清理策略保证数据持久化
- ❌ 缓存故障可能导致数据丢失

**适用场景**:
- 写入性能要求高的场景
- 可以容忍一定数据丢失风险的场景

**代码路径**: `engine_wb.c`

### 2.3 Write-Around (WA) - 写旁路模式

**工作原理**:
- **读操作**: 
  - Cache Hit: 从缓存读取
  - Cache Miss: 从核心读取，并写入缓存
- **写操作**: 
  - 直接写入核心，不写入缓存
  - 如果缓存中有数据，先更新缓存

**特点**:
- ✅ 避免写入污染缓存
- ✅ 适合顺序写入场景
- ❌ 写入后首次读取会miss

**适用场景**:
- 顺序写入为主的工作负载
- 写入数据访问模式不规律

**代码路径**: `engine_wa.c`

### 2.4 Write-Invalidate (WI) - 写无效模式

**工作原理**:
- **读操作**: 
  - Cache Hit: 从缓存读取
  - Cache Miss: 从核心读取，并写入缓存
- **写操作**: 
  - 直接写入核心
  - 如果缓存中有数据，使其无效

**特点**:
- ✅ 简单高效
- ✅ 避免缓存污染
- ❌ 写入后首次读取会miss

**适用场景**:
- 写入后很少再次读取的场景

**代码路径**: `engine_wi.c`

### 2.5 Write-Only (WO) - 仅写模式

**工作原理**:
- **读操作**: 
  - 总是从核心读取（不缓存）
- **写操作**: 
  - 写入缓存（Write-Back方式）

**特点**:
- ✅ 只缓存写入数据
- ✅ 适合写入密集型工作负载
- ❌ 读取性能无提升

**适用场景**:
- 写入密集型应用
- 读取模式不可预测

**代码路径**: `engine_wo.c`

### 2.6 Pass-Through (PT) - 透传模式

**工作原理**:
- **所有操作**: 
  - 直接透传到核心，不经过缓存

**特点**:
- ✅ 完全绕过缓存
- ✅ 用于故障恢复或测试
- ❌ 无缓存加速效果

**适用场景**:
- 缓存故障时的降级模式
- 性能测试基准
- 特定I/O类别的透传

**代码路径**: `engine_pt.c`

### 2.7 缓存模式选择

| 模式 | 读性能 | 写性能 | 一致性 | 适用场景 |
|------|--------|--------|--------|----------|
| **WT** | 高 | 低 | 最高 | 数据库、关键应用 |
| **WB** | 高 | 最高 | 中等 | 高性能计算、临时数据 |
| **WA** | 高 | 中 | 高 | 顺序写入、日志 |
| **WI** | 高 | 中 | 高 | 写入后不读 |
| **WO** | 无 | 高 | 中等 | 写入密集型 |
| **PT** | 无 | 无 | 高 | 故障恢复 |

## 3. 核心数据结构

### 3.1 Cache Line (缓存行)

**定义**:
- 缓存的基本单位
- 大小可配置：4KB、8KB、16KB、32KB、64KB
- 每个缓存行对应核心设备的一个地址范围

**元数据**:
- **状态位**: 有效、脏、锁定等
- **核心ID**: 属于哪个核心
- **地址映射**: 核心地址到缓存地址的映射
- **淘汰信息**: LRU链表节点等

### 3.2 Metadata (元数据)

**存储位置**:
- **Volatile**: 存储在内存中（重启后丢失）
- **Persistent**: 存储在缓存设备上（持久化）

**布局方式**:
- **Striping**: 条带化布局（默认）
- **Sequential**: 顺序布局

**元数据结构**:
```c
// 每个缓存行的元数据
struct ocf_metadata_map {
    ocf_cache_line_t cache_line;  // 缓存行地址
    ocf_core_id_t core_id;         // 核心ID
    uint64_t core_line;            // 核心行地址
    uint8_t status;                // 状态位
    // ...
};
```

### 3.3 Request (请求)

**结构**:
```c
struct ocf_request {
    ocf_cache_t cache;              // 缓存实例
    ocf_core_t core;                // 核心实例
    uint64_t byte_position;         // 字节位置
    uint32_t byte_length;           // 字节长度
    uint32_t rw;                    // 读/写
    ocf_req_cache_mode_t cache_mode; // 缓存模式
    struct ocf_io_if *io_if;        // I/O接口
    // ...
};
```

## 4. I/O处理流程

### 4.1 请求处理流程

```
用户提交I/O请求
    ↓
ocf_core_submit_io()
    ↓
ocf_io_handle() - 创建ocf_request
    ↓
ocf_resolve_effective_cache_mode() - 确定缓存模式
    ↓
ocf_engine_hndl_req() - 处理请求
    ↓
根据缓存模式选择引擎:
  - engine_wt: Write-Through
  - engine_wb: Write-Back
  - engine_wa: Write-Around
  - engine_wi: Write-Invalidate
  - engine_wo: Write-Only
  - engine_pt: Pass-Through
    ↓
执行I/O操作:
  1. 查找缓存行 (lookup)
  2. 处理命中/未命中
  3. 更新元数据
  4. 提交I/O到缓存/核心
    ↓
完成回调 (ocf_end_io_t)
```

### 4.2 缓存查找 (Lookup)

**步骤**:
1. **计算缓存行地址**: `cache_line = addr / cache_line_size`
2. **查找元数据**: 检查该缓存行是否有效
3. **验证核心ID**: 确认是否属于当前核心
4. **检查状态**: 有效、脏、锁定等

**结果**:
- **HIT**: 数据在缓存中
- **MISS**: 数据不在缓存中
- **MAPPED**: 缓存行已映射但数据无效

### 4.3 写回模式详细流程

以Write-Back模式为例：

```
写入请求到达
    ↓
ocf_write_wb()
    ↓
获取缓存行锁 (WR lock)
    ↓
检查缓存状态:
  - HIT: 更新缓存，标记dirty
  - MISS: 分配缓存行，写入数据，标记dirty
    ↓
提交写入到缓存设备
    ↓
更新元数据 (标记dirty)
    ↓
完成回调
    ↓
(后台) 清理策略将dirty数据写入核心
```

**关键点**:
- 写入只到缓存，不立即写入核心
- 标记为dirty，由清理策略异步写入
- 需要空间时，先清理dirty数据

## 5. 淘汰策略 (Eviction Policy)

### 5.1 LRU (Least Recently Used)

**原理**:
- 维护LRU链表
- 最近使用的缓存行在链表头部
- 需要空间时，从链表尾部淘汰

**实现**:
- 每个缓存行维护LRU链表节点
- 访问时移动到链表头部
- 淘汰时从尾部移除

**数据结构**:
```c
struct lru_eviction_policy {
    struct list_head hot_list;      // 热数据链表
    struct list_head cold_list;     // 冷数据链表
    // ...
};
```

### 5.2 淘汰流程

```
需要分配缓存行
    ↓
检查空闲列表 (freelist)
    ↓
如果空闲列表为空:
    ↓
调用淘汰策略 (eviction)
    ↓
从LRU链表尾部选择缓存行
    ↓
如果缓存行是dirty:
    ↓
先清理到核心 (flush)
    ↓
使缓存行无效
    ↓
加入空闲列表
    ↓
分配给新请求
```

## 6. 清理策略 (Cleaning Policy)

### 6.1 NOP (No Operation)

**特点**:
- 不进行后台清理
- 只在淘汰或停止时清理
- 最简单，无额外开销

**适用场景**:
- 写入量小的场景
- 对清理延迟不敏感

### 6.2 ALRU (Approximate LRU)

**特点**:
- 基于LRU的近似算法
- 在I/O空闲时清理
- 清理最久未使用的dirty数据

**参数**:
- **Wake Up Time**: 清理线程唤醒间隔
- **Stale Buffer Time**: 数据被认为是stale的时间
- **Activity Threshold**: 触发清理的活跃度阈值

**适用场景**:
- 一般工作负载
- 需要平衡性能和一致性

### 6.3 ACP (Adaptive Cleaning Policy)

**特点**:
- 自适应清理策略
- 减少核心设备寻道距离
- 与I/O并发运行

**优化**:
- 按地址顺序清理
- 减少磁盘寻道
- 提高清理效率

**适用场景**:
- 机械硬盘后端
- 需要优化寻道的场景

---

## 7. 提升策略 (Promotion Policy)

### 7.1 Always (总是提升)

**特点**:
- 所有读取都提升到缓存
- 默认策略
- 简单高效

### 7.2 NHIT (N次命中)

**特点**:
- 数据需要被访问N次后才提升到缓存
- 减少缓存污染
- 提高缓存命中率

**参数**:
- **Hit Count Threshold**: 命中次数阈值

**适用场景**:
- 访问模式不规律
- 需要过滤冷数据

## 8. 元数据管理

### 8.1 元数据布局

#### 8.1.1 Striping布局

**特点**:
- 元数据分散在缓存设备上
- 与数据交错存储
- 减少元数据集中访问

**优势**:
- 元数据访问分散
- 减少热点

#### 8.1.2 Sequential布局

**特点**:
- 元数据集中在缓存设备开头
- 数据紧随其后
- 简单直接

**优势**:
- 元数据访问集中
- 易于管理

### 8.2 元数据持久化

**持久化时机**:
- 缓存停止时
- 定期保存（可配置）
- 手动保存 (`ocf_mngt_cache_save`)

**恢复**:
- 启动时加载 (`ocf_mngt_cache_load`)
- 验证元数据完整性
- 恢复缓存状态

## 9. 并发控制

### 9.1 缓存行锁

**锁类型**:
- **RD (Read Lock)**: 读锁，允许多个读操作
- **WR (Write Lock)**: 写锁，独占访问

**锁粒度**:
- 每个缓存行一个锁
- 细粒度并发控制

### 9.2 元数据锁

**作用**:
- 保护元数据一致性
- 防止并发修改

**实现**:
- 分区锁（Partition Lock）
- 减少锁竞争

## 10. I/O类 (IO Class)

### 10.1 概念

I/O类用于将缓存空间划分为不同的分区，每个分区可以有不同的：
- **缓存模式**: 可以为不同I/O类设置不同模式
- **优先级**: 影响淘汰顺序
- **大小限制**: 最小/最大缓存空间

### 10.2 配置

```c
struct ocf_mngt_io_class_config {
    uint32_t class_id;              // I/O类ID (0-32)
    const char *name;               // I/O类名称
    int16_t prio;                   // 优先级 (-1=固定, 0-255)
    ocf_cache_mode_t cache_mode;    // 缓存模式
    uint32_t min_size;              // 最小大小
    uint32_t max_size;              // 最大大小
};
```

### 10.3 使用场景

- **数据库**: 索引数据使用高优先级I/O类
- **日志**: 日志数据使用低优先级或Write-Around模式
- **临时文件**: 使用Write-Only模式

## 11. 顺序截断 (Sequential Cutoff)

### 11.1 功能

**目的**: 避免顺序I/O污染缓存

**策略**:
- **Always**: 总是截断顺序I/O
- **Full**: 缓存满时截断
- **Never**: 不截断

### 11.2 工作原理

```
检测到顺序I/O模式
    ↓
检查顺序截断策略
    ↓
如果满足截断条件:
    ↓
直接透传到核心 (Pass-Through)
    ↓
不写入缓存
```

**优势**:
- 避免顺序写入占用缓存空间
- 提高随机I/O的缓存命中率

---

## 12. 快速路径 (Fast Path)

### 12.1 概念

**目的**: 优化常见场景，减少开销

**条件**:
- 请求对齐到缓存行边界
- 缓存行已锁定
- 无元数据更新需求

### 12.2 优化效果

- 减少锁竞争
- 减少函数调用开销
- 提高I/O性能

---

## 13. 故障处理

### 13.1 Fallback Pass-Through

**触发条件**:
- I/O错误达到阈值
- 自动切换到Pass-Through模式

**目的**:
- 保证系统可用性
- 避免缓存故障影响业务

### 13.2 错误计数

```c
// 错误计数器
env_atomic_t fallback_pt_error_counter;

// 错误阈值
uint32_t fallback_pt_error_threshold;
```

## 15. 设计原理总结

### 15.1 异步I/O模型

**原理**:
- 所有I/O操作都是异步的
- 通过队列和回调实现
- 支持高并发

**优势**:
- 非阻塞I/O
- 高吞吐量
- 低延迟

### 15.2 环境抽象

**原理**:
- 通过环境抽象层隔离平台差异
- 用户提供平台特定实现
- OCF核心代码平台无关

**接口**:
- 内存分配/释放
- 锁/互斥量
- 线程/工作队列
- I/O操作

### 15.3 模块化设计

**模块**:
- **引擎模块**: 不同缓存模式的实现
- **元数据模块**: 元数据管理
- **淘汰模块**: 淘汰策略
- **清理模块**: 清理策略
- **提升模块**: 提升策略

**优势**:
- 易于扩展
- 易于测试
- 代码清晰

### 15.4 性能优化

**优化技术**:
1. **快速路径**: 优化常见场景
2. **批量操作**: 批量处理I/O
3. **元数据缓存**: 缓存热点元数据
4. **锁优化**: 细粒度锁，减少竞争
5. **顺序检测**: 避免顺序I/O污染缓存

## 16. 使用示例

### 16.1 基本使用流程

```c
// 1. 初始化OCF上下文
ocf_ctx_t ctx;
ctx_init(&ctx);

// 2. 启动缓存
ocf_cache_t cache;
struct ocf_mngt_cache_config cache_cfg;
ocf_mngt_cache_config_set_default(&cache_cfg);
cache_cfg.name = "cache1";
ocf_mngt_cache_start(ctx, &cache, &cache_cfg);

// 3. 附加缓存设备
struct ocf_mngt_cache_device_config device_cfg;
ocf_mngt_cache_device_config_set_default(&device_cfg);
ocf_uuid_set_str(&device_cfg.uuid, "/dev/nvme0n1");
device_cfg.volume_type = VOL_TYPE_BLOCK;
ocf_mngt_cache_attach(cache, &device_cfg, complete, priv);

// 4. 添加核心
ocf_core_t core;
struct ocf_mngt_core_config core_cfg;
ocf_mngt_core_config_set_default(&core_cfg);
strcpy(core_cfg.name, "core1");
ocf_uuid_set_str(&core_cfg.uuid, "/dev/sda");
core_cfg.volume_type = VOL_TYPE_BLOCK;
ocf_mngt_cache_add_core(cache, &core_cfg, add_core_complete, priv);

// 5. 提交I/O
struct ocf_io *io = ocf_core_new_io(core, queue, addr, len, OCF_WRITE, 0, 0);
ocf_io_set_data(io, data, 0);
ocf_io_set_cmpl(io, priv1, priv2, complete_callback);
ocf_core_submit_io(io);

// 6. 停止缓存
ocf_mngt_cache_stop(cache, stop_complete, priv);
```

### 16.2 配置缓存模式

// 设置清理策略

## 17. 性能特点

### 17.1 延迟特性

- **缓存命中读取**: < 10μs (取决于存储介质)
- **缓存未命中读取**: 核心设备延迟 + 缓存写入延迟
- **Write-Through写入**: 缓存延迟 + 核心延迟
- **Write-Back写入**: 仅缓存延迟 (< 10μs)

### 17.2 吞吐量

- **读取吞吐量**: 可达数GB/s（取决于缓存设备）
- **写入吞吐量**: 
  - Write-Back: 接近缓存设备吞吐量
  - Write-Through: 受限于核心设备吞吐量

### 17.3 扩展性

- **最大核心数**: 4096个（可配置）
- **最大I/O类数**: 33个（可配置）
- **缓存大小**: 最小20MB，无上限
